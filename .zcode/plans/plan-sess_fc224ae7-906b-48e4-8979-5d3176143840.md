# Профессиональный ИИ для бота дурака — Шаги 2 + 3 + интерфейс NN

## Зафиксированные решения
- **Объём**: Шаг 2 (битборды + абсолютная память) + Шаг 3 (ISMCTS для фазы с колодой + Minimax α-β для эндшпиля) + **интерфейс NN как заглушка** (полиморфный `PolicyValueNet` с `RandomNet` по умолчанию; реальный ONNX/TorchScript добавляется позже без переписывания ISMCTS).
- **Режим**: интерактивный советник с таймаутом. В UI три уровня силы: «быстро» (~0.5с), «норма» (~2с), «глубоко» (~10с). Пул потоков гоняет playout до таймаута.
- **Симулятор**: единый движок-симулятор `MatchState` поверх существующих `rules`. Один источник истины для UI / ISMCTS / минимакса / будущего pybind11.

## Честное ограничение по NN
Нейросеть гроссмейстерского уровня требует обучения (миллионы партий self-play на вашей 4060 Ti). Сейчас закладывается только **архитектура и точка интеграции** — `RandomNet` вернёт случайные policy/value, поэтому ISMCTS будет работать на чистой статистике playout (что само по себе заметно сильнее эвристики). Запуск self-play — отдельная итерация.

## Часть А — Битборды и абсолютная память (Шаг 2)

### `engine/src/bitboard.h / .cpp` (новый)
- `using CardMask = uint64_t;` — 36 карт в одном машинном слове (младшие 36 бит).
- Кодирование: `cardIndex(Rank, Suit) = (suitIndex*9 + rankIndex)`, `rankIndex = rank-6 ∈ [0,8]`.
- Операции (inline, ~1 такт): `add`, `remove`, `has`, `popCount` (через `__popcnt64`/`std::popcount`), `isEmpty`, `count`.
- Конверсии `Card↔CardMask`, `vector<Card>↔CardMask`, сериализация в/из IPC (`{r,s}`-массивы).

### `engine/src/knowledge.h / .cpp` (новый) — «Абсолютная память»
Структура, которую бот держит по ходу партии:
```
struct Knowledge {
    CardMask myHand;          // 100% известно
    Suit     trump;
    CardMask discard;         // каждая ушедшая в отбой карта (память бота)
    CardMask oppKnownTaken;   // карты, которые соперник ЗАБРАЛ со стола (мы это видели)
    int      oppHandCount;    // сколько сейчас карт у соперника
    int      deckRemaining;   // сколько в колоде
    // Неизвестный пул = все 36 − myHand − discard − oppKnownTaken − (deckRemaining-?)
    CardMask unknownPool() const;   // множество карт, распределение которых неизвестно
}
```
- `unknownPool` = вычитание масок: всё, что не в моей руке/бите/известно-взятом соперником.
- Из `unknownPool` ISMCTS семплирует: `oppHandCount` карт → сопернику, остаток → колода (порядок случаен, т.к. не влияет на правила добора по счёту).

## Часть Б — Единый движок-симулятор (база для ИИ)

### `engine/src/match.h / .cpp` (новый) — `MatchState`
Полная внутренняя модель партии для симуляций вперёд:
```
struct MatchState {
    CardMask hands[2];        // [0]=бот, [1]=соперник (в playout обе известны!)
    CardMask deck;            // в симуляции колода конкретна
    Suit     trump;
    CardMask discard;
    TablePair table[6]; int tableLen;
    int pairsLimit;           // 5 в первом коне, иначе 6
    int  attacker, defender, turn;
    Phase phase;
    bool firstTrick, transferEnabled, flashEnabled;
    int  deckRemaining;       // счётчик для добора
}
```
- `applyMove(MatchState&, Move)` — детерминированный переход: атака/подкидывание/защита/перевод/взять/бито, включая:
  - **конец кона**: «бито» → стол в `discard`, ход защищавшемуся; «беру» → стол в руку взявшего, ход атакующему;
  - **накидывание вдогонку** при «беру»;
  - **добор до 6**: первым атаковавший, затем защищавшийся (после перевода — последний атаковавший); из `deck` снимаем старшие биты по условленной схеме.
- `deal(...)` / `drawHands(...)` — инициализация и добор.
- `isGameOver()` — колода пуста И у кого-то 0 карт (или после полного завершения).
- `winner()` — 0/1/ничья.
- **Переиспользование правил**: функции валидации/генерации ходов из `rules.cpp` адаптируются для работы с битмасочной рукой (без копирования в `vector<Card>` в горячем цикле — критично для скорости playout).

### Оптимизация playout-генератора
В `rules` добавить «быстрые» варианты, оперирующие `CardMask` напрямую:
- `legalMovesMask(const MatchState&)` → `vector<Move>` без аллокаций лишних промежуточных структур.
- Лёгкий `defaultPolicy(MatchState&)` для playout: равномерно-случайный выбор из легальных ходов с лёгким предпочтением «бить младшей достаточной» (уменьшает шум playout).

## Часть В — ISMCTS (Шаг 3, фаза с колодой)

### `engine/src/ismcts.h / .cpp` (новый)
Алгоритм **Information Set MCTS** (Cowling et al., 2012) — стандарт для игр со скрытой информацией:
- **Information Set (IS)** = множество детерминизаций (конкретных рук соперника + колоды), совместимых с наблюдениями бота.
- Каждый узел дерева = `{ information set, дети по ходам, N (visit count), W (выигрыши) }`.
- Цикл одной симуляции:
  1. **Determine** — из `Knowledge.unknownPool()` случайно раздать сопернику `oppHandCount` карт, остаток — в колоду → детерминизация `MatchState`.
  2. **Select** по UCB1 до нерасширенного узла.
  3. **Expand** — добавить детей по легальным ходам в детерминизации.
  4. **Rollout** — `defaultPolicy` до конца партии (используется единый симулятор).
  5. **Backprop** — обновить `N/W` по всем узлам пути.
- Выбор хода = корневой ребёнок с максимальным `N` (робастный выбор).
- **Параметр решающего**: если NN доступен (не `RandomNet`), rollout обрывается на фиксированной глубине и оценка листа берётся из `net.value(...)`, а prior ходов — из `net.policy(...)` (PUCT вместо UCB1). Это та точка, где сеть ускоряет поиск.

### Многопоток
- `ThreadPool` (новый, `engine/src/threadpool.h/.cpp`) на `std::thread` + `std::mutex`. 7500F = 12 потоков SMT.
- Корневое дерево одно, защищено спин-локом; каждый воркер крутит цикл determine→select→rollout→backprop до общего флага «время вышло» (атомик `std::atomic<bool>`).
- Время задаётся таймаутом силы (быстро/норма/глубоко).

## Часть Г — Minimax α-β (Шаг 3, эндшпиль)

### `engine/src/endgame.h / .cpp` (новый)
Срабатывает, когда `deckRemaining == 0`:
- Противник = «все карты не в моей руке и не в бите» → известны полностью → **perfect information**.
- `int minimax(MatchState, depth, α, β, maximizing)` с α-бета отсечением.
- `Move bestEndgameMove(const MatchState&)` → перебор по своим ходам, выбор максимизирующего.
- Итеративное углубление + таблица перестановок (transposition table на `unordered_map<uint64_t,TTEntry>`) — эндшпили в дураке невелики по глубине, часто находится форсированный выигрыш.
- Узлы упорядочены: сначала перевод/младшая защита (повышает отсечения).
- Ограничение по времени (тот же таймаут силы) на случай слишком глубокого дерева — тогда берём лучший по глубине.

## Часть Е — Склейка в боте и UI

### `bot.cpp` — переработка `decide()`
```
Move decide(const GameState& s, const Knowledge& k, Strength strength, PolicyValueNet* net) {
    if (s.deck.remaining == 0)
        return endgame::bestEndgameMove(toMatchState(s, /*oppHand=*/k.unknownPool fully dealt));
    else
        return ismcts::bestMove(k, strength, net, threadPool);
}
```
- Состояние из IPC теперь обогащается полями памяти (`discard`, `oppKnownTaken`), UI их собирает.

### IPC — расширение протокола
В `setState` добавятся: `discard: [...]`, `oppTaken: [...]`. `decide` возвращает `{action, rank, suit, target, reason, stats:{playouts, winrate, timeMs, mode}}`.
Новый параметр силы: `{"cmd":"decide","strength":"normal"}` (`fast|normal|deep`).

### UI (`src/`)
- Поле «Карты в бите» (множественный выбор из тех же 36) — абсолютная память бота.
- Поле «Соперник забрал» (карты, которые он взял со стола в руку).
- Селектор силы: «быстро / норма / глубоко».
- В блоке совета показывать статистику: `mode=ISMCTS, playouts=28431, winrate=0.62, 2.0s` или `mode=Endgame, depth=9, solved`.
